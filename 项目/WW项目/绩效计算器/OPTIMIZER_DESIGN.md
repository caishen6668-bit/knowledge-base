# 人员调配优化算法 — 设计方案

> 版本：v1.0（2026-07-22）
> 状态：待确认，确认后开发 optimizer.py

---

## 一、问题定义

### 1.1 场景

每天运行绩效计算后，系统根据当日排班（谁来上班）、各阶段人员需求（每个队列要多少人）、
能力画像（每人每阶段的 AbilityScore），输出一份**人员调配建议**。

调配建议 = 将每位在岗员工分配到具体队列/阶段。

### 1.2 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 在岗人员列表 | scheduler_api（排班系统） | 当天上班的员工，含休息排除 |
| 阶段需求 | 待定义（手工配置 / 排班系统 / 规则） | 每个队列需要多少人 |
| 能力画像 | ability.py → ability_database.xlsx | 每人每阶段的 AbilityScore / Confidence / Experience |
| 昨日安排 | ability.py → LastWorkDate / 每日明细 | 每人昨天在哪个队列 |
| 调岗历史 | history.py（后续） | 最近调岗记录，避免短期内反复调动 |

### 1.3 输出

| 输出 | 说明 |
|------|------|
| 调配方案 | {staff_id → queue} 映射 |
| 调岗清单 | 哪些人从哪个队列调到了哪个队列 |
| 方案评分 | 整体能力总分 + 调岗人数 + 稳定性指标 |
| 未满足需求 | 如果无法满足所有需求，列出缺口 |

## 二、约束与目标

### 2.1 硬约束（不可违反）

| 编号 | 约束 | 实现方式 |
|------|------|------|
| **C1** | 同一人不能分配到多个队列 | 每人最多一个岗位 |
| **C2** | 休息人员不参与分配 | 排班系统过滤，不在 S 中 |
| **C3** | 所有岗位必须填满 | 需求必须满足（不足时报异常） |
| **C4** | 能力不足者不能上岗 | 最低能力阈值（可选，待业务确认） |

### 2.2 软目标（按优先级排序）

| 优先级 | 目标 | 量化方式 |
|:---:|------|------|
| P1 | 满足所有阶段需求 | 所有 demand[q] 必须填满 |
| P2 | 尽量保持昨天岗位 | Maximize stability_count |
| P3 | 调岗人数最少 | Minimize transfer_count |
| P4 | 整体能力最高 | Maximize Σ AbilityScore[staff][assigned_queue] |

### 2.3 目标优先级说明

当目标冲突时（如：保持昨天岗位 vs 提升整体能力），优先级裁决：

```
P1 >> P2 >> P3 >> P4
```

即：先保证需求满足；在需求满足的前提下尽量减少岗位变动；
在同样变动次数下，选择能力更高的方案。

## 三、跨阶段调岗收益计算

### 3.1 核心问题

能力最高的人集中在某个阶段（如 D0），但其他阶段（如 S1）缺人。
如果把一个 D0 高能力者调到 S1：

- 他可能在 S1 能力一般，不如留在 D0
- 但 S1 缺人，必须有一个人去
- **问题**：调谁去 S1 对系统整体能力的损失最小？

### 3.2 比较优势原理

不能简单地选"在目标阶段能力最高的人"去调岗。而应该选"在原阶段损失最小、在目标阶段收益最大"的人。

**示例：**

```
          D0能力   S1能力
员工A     0.40     0.14
员工B     0.22     0.10

昨日：A→D0, B→D0
需求：D0=1, S1=1（必须有一人调去 S1）
```

| 方案 | D0 | S1 | 总分 | 调岗 |
|------|:--:|:--:|:----:|:---:|
| A调S1, B留D0 | 0.22 | 0.14 | 0.36 | 1 |
| B调S1, A留D0 | 0.40 | 0.10 | **0.50** | 1 |

→ 调 B 更优。虽然 A 的 S1 能力比 B 高（0.14 > 0.10），但 A 留在 D0 的贡献远大于 B。

### 3.3 个人调岗收益

个人调岗收益衡量：把某人从 原阶段 调到 目标阶段，对系统的影响。

```
个人调岗收益 = AbilityScore[P][目标] - AbilityScore[P][原]
```

- 正值 → 调岗后此人贡献更大
- 负值 → 调岗后此人贡献变小（但不调可能无法满足需求）

### 3.4 配对交换收益

最优情况是一对一交换：P 从 X 调到 Y，Q 从 Y 调到 X。

```
配对收益 = (Ability[P][Y] + Ability[Q][X]) - (Ability[P][X] + Ability[Q][Y])
```

- 正值 → 交换后系统总能力提升
- 负值 → 交换后系统总能力下降

**例（续上例）：**

```
A留在D0, B调到S1:
配对收益 = (A_D0 + B_S1) - (A_D0 + B_D0)
         = (0.40 + 0.10) - (0.40 + 0.22)
         = -0.12  → 负值，但这是强制的（需求驱动）

A调到S1, B留在D0:
配对收益 = (A_S1 + B_D0) - (A_D0 + B_D0)
         = (0.14 + 0.22) - (0.40 + 0.22)
         = -0.26  → 更差
```

选择负值最小的方案（损失最小）。

### 3.5 多对多置换收益

当 2 个以上的人需要调整时，问题退化为线性分配问题。每个 (人, 岗位) 产生一个成本：

```
Cost(P, q) = -AbilityScore[P][q] + TransferPenalty(P, q)

TransferPenalty(P, q) = 0          如果 q == 昨日队列
                       = TRANSFER_COST  如果 q != 昨日队列
```

TRANSFER_COST 越大 → 越不愿调岗；越小 → 越重视能力匹配。

## 四、算法方案

### 4.1 方案 A：Stability-First Greedy（稳定优先贪心）

**核心理念**：除非不得不动，否则人人原地不动。

**流程：**

```
Step 1: 锁定昨日岗位
  FOR EACH 在岗员工 P:
    IF P.昨日队列 仍有需求:
      分配 P → P.昨日队列
      demand[P.昨日队列] -= 1

Step 2: 填补缺口
  FOR EACH 未满足队列 q (demand[q] > 0):
    从"未分配"员工中，按 AbilityScore[q] 降序排列
    FOR EACH 候选 (按能力从高到低):
      IF 候选尚未分配:
        分配 候选 → q
        demand[q] -= 1
        IF demand[q] == 0: BREAK

Step 3: 强制调岗（如果还有缺口）
  FOR EACH 未满足队列 q:
    FOR EACH 已分配到其他队列的员工 P:
      计算 个人调岗收益 = Ability[P][q] - Ability[P][P.当前队列]
    选择 收益最高 的 P
    IF 调岗后 P.原队列 出现缺口 → 递归填补
    执行调岗

Step 4: 输出方案
```

**优点：**
- 调岗人数最少
- 逻辑简单、可解释性强
- 员工体验好（不频繁变动）

**缺点：**
- 整体能力可能不是最优
- 强制调岗步骤逻辑复杂（递归填补）

**适用场景：** 团队稳定期，需求变动不大。

---

### 4.2 方案 B：Ability-First Global（能力优先全局优化）

**核心理念**：把所有 (人, 岗位) 当作一个全局最优化问题求解。

**流程：**

```
Step 1: 构建成本矩阵
  n = 在岗员工数
  m = 岗位总数（∑ demand[q]）

  Cost[i][j] = -AbilityScore[员工_i][岗位_j的队列]
              + (0 如果 岗位_j == 员工_i.昨日队列 else TRANSFER_COST)

Step 2: 求解最优分配
  使用 匈牙利算法（Hungarian Algorithm）或 最小费用流
  找到使总成本最小的分配方案

Step 3: 输出方案
  解析分配结果 → 与昨日对比 → 生成调岗清单
```

**匈牙利算法简介：**
- 经典 O(n³) 分配算法
- 输入：n×n 成本矩阵
- 输出：每个员工的唯一岗位，使总成本最小
- 如果 m > n（岗位多于员工），用虚拟员工补齐；如果 n > m，用虚拟岗位补齐

**优点：**
- 数学上最优（在给定的 Cost 函数下）
- 自动权衡能力 vs 稳定性
- 调岗收益计算自然融入

**缺点：**
- 复杂度较高（但 n 通常 ≤ 50，完全可以接受）
- TRANSFER_COST 参数需要调优
- 结果可能对参数敏感

**适用场景：** 需要最优方案，团队规模可控。

---

### 4.3 方案 C：Hybrid Gain-Threshold（混合收益阈值）

**核心理念**：先默认稳定，只执行收益超过阈值的调岗。

**流程：**

```
Step 1: 默认分配（同方案A Step 1）
  所有人先留在昨天岗位

Step 2: 识别缺口和富余
  gaps = {q: demand[q] 未满足的}
  surplus = {q: 当前分配人数 > demand[q] 的}

Step 3: 计算所有可能的调岗收益
  FOR EACH (富余队列 X, 缺口队列 Y):
    FOR EACH 在 X 中的人员 P:
      FOR EACH 在 Y 中的人员 Q（或空位）:
        计算 配对交换收益 = Ability[P][Y] + Ability[Q][X]
                           - Ability[P][X] - Ability[Q][Y]

Step 4: 按收益排序，超过阈值的执行
  按 配对收益 降序排列
  FOR EACH (收益 > MIN_TRANSFER_GAIN):
    执行交换
    更新 gaps 和 surplus
    IF gaps 为空: BREAK

Step 5: 填补剩余缺口
  若还有缺口 → 归入方案A的 Step 2-3
```

**阈值设计：**

| 阈值 | 含义 |
|------|------|
| GAIN > 0 | 只有真正提升能力才调 |
| GAIN > -0.02 | 允许轻微损失以保满足需求 |
| GAIN > -∞ | 必须满足需求（等同于强制调岗） |

**优点：**
- 结合了稳定和优化的优点
- 阈值可配置，灵活调整
- 调岗决策有据可查（"因为收益 0.05 > 0.02"）

**缺点：**
- 阈值需要业务验证
- 介于贪心和全局最优之间，要么有未优化的遗憾

**适用场景：** 大多数日常场景，需要灵活平衡。

---

### 4.4 方案 D：Multi-Objective Scoring（多目标评分）

**核心理念**：为每个 (人, 岗位) 打综合分，贪心分配。

**流程：**

```
Step 1: 为每对 (员工 P, 岗位 q) 计算综合评分

  Score(P, q) = w₁ × Ability_norm(P, q)
              + w₂ × Stability(P, q)
              + w₃ × Confidence_norm(P, q)
              + w₄ × Experience_norm(P, q)

  其中：
    Ability_norm    = AbilityScore[P][q] / max_ability_in_stage[q]  (归一化到 [0,1])
    Stability(P,q)  = 1 if q == P.昨日队列 else 0
    Confidence_norm = Confidence[P][q] / 100
    Experience_norm = min(Experience[P][q] / MAX_EXPERIENCE, 1)

Step 2: 分配
  按岗位空缺数展开：m = Σ demand[q] 个岗位
  每个员工对每个岗位有 Score(P, j)
  贪心分配：每次选择 Score 最高的 (P, j) 对
  分配后移除 P，继续直到所有岗位填满

Step 3: 检查约束
  IF 有岗位未填满:
    降低权重 → 放宽要求 → 重新分配
```

**权重配置：**

| 场景 | w₁ (能力) | w₂ (稳定) | w₃ (置信) | w₄ (经验) |
|------|:---:|:---:|:---:|:---:|
| 日常运营 | 40 | 40 | 10 | 10 |
| 攻坚期（追求业绩） | 60 | 20 | 10 | 10 |
| 稳定期（减少变动） | 20 | 60 | 10 | 10 |
| 新人培训期 | 20 | 30 | 10 | 40 |

**优点：**
- 灵活：权重按业务场景切换
- 透明：每个决策可解释（"因为他能力 + 稳定综合最高"）
- 不需要调 TRANSFER_COST（更直观）

**缺点：**
- 贪心分配非全局最优
- 权重需要经验调优

**适用场景：** 需要灵活调整策略的团队。

## 五、方案对比

| 维度 | 方案A 稳定优先 | 方案B 全局最优 | 方案C 收益阈值 | 方案D 多目标 |
|------|:---:|:---:|:---:|:---:|
| 算法复杂度 | O(n²) | O(n³) | O(n²) | O(n²·m) |
| 全局最优 | ❌ | ✅ | ❌（近似） | ❌ |
| 调岗最少 | ✅ | ❌（取决于参数） | ✅（有阈值） | 取决于权重 |
| 可解释性 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 参数数量 | 0 | 1 (TRANSFER_COST) | 2 (阈值 + 成本) | 4 (权重) |
| 应对需求波动 | 一般 | 好 | 好 | 好 |
| 实现难度 | 低 | 中 | 中 | 低 |

```
  │     ├─ 调配方案: {staff_id → queue}

  │     ├─ 方案评分: total_ability, transfer_count, stability_pct
  │     └─ 未满足需求: []（正常情况下为空）
  │
  └─► 返回调配结果

```

## 八、决策边界与风险

### 8.1 能力不足

**场景：** 某阶段所有候选人的 AbilityScore 都很低（如 < 0.05）。

**处理：**
- 如果设置了最低阈值 → 拒绝分配，输出警报
- 如果未设置阈值 → 选能力最高的（即使很低），输出警告

### 8.2 人员不足

**场景：** 在岗人数 < 总需求。

**处理：**
- 输出警告：哪些队列缺多少人
- 优先满足高优先级队列（需定义队列优先级）
- 生成"缺口报告"供管理者决策（加班？外援？）

### 8.3 新员工（无历史数据）

**场景：** 新员工在 ability_database 中无记录。

**处理：**
- AbilityScore 默认 = 0（或使用队列平均值）
- Confidence = 0
- Experience = 0
- 输出警告：新员工缺少能力数据

### 8.4 并列最优

**场景：** 两个员工的评分完全相同。

**处理：**
- 优先选 Experience 高的
- 其次选 Confidence 高的
- 最后按工号字典序（确定性）

## 九、建议

### 推荐实施路线

| 阶段 | 内容 |
|------|------|
| Phase 1 | **方案 D（多目标评分）** 作为默认算法 — 灵活、可解释、易调参 |
| Phase 2 | 同时实现 **方案 A** 作为降级方案 — 需求突变时一键回退"稳定模式" |
| Phase 3 | 积累数据后，将 **方案 B（匈牙利算法）** 作为可选高级模式 |
| Phase 4 | 方案 C 的阈值逻辑可作为方案 D 的补充（"收益 > X 才推荐调岗"） |

### 理由

1. **方案 D** 对业务方最友好 — 权重含义直观，可以开会直接调
2. **方案 A** 是最安全的 fallback — 无论如何不会出大乱
3. **方案 B** 是最优解 — 数据积累后可以验证其他方案离最优有多远
4. 四种方案可以共存于 optimizer.py 中，通过 `config.py` 或命令行参数切换
